# sync-memory

このセッションの会話内容（transcript）を解析し、以下の記録・更新を**すべて実行**してください。

## 実行手順

### 0. 既存データの確認（重複防止）

**最初に必ず実行する。** 前回のsync-memoryやセッション中の手動記録で既に記録されている可能性があるため、既存データを確認してから記録する。

**アクション:**
- セッション中に使ったトピックの `get_decisions(topic_id=...)` と `get_logs(topic_id=...)` を実行し、直近の決定事項・ログを取得
- `get_activities(status="active")` で未完了アクティビティを取得（pending + in_progress）
- 取得した既存データと会話内容を照合し、**既に記録済みの内容は記録しない**

**重複の判定基準:**
- 同じ意味・内容のdecisionが既にあれば記録しない（完全一致でなくても意味が同じならスキップ）
- 同じ意味・内容のlogが既にあれば記録しない
- 同じタイトル・目的のアクティビティが既にあれば記録しない
- 同じテーマのtopicが既にあれば記録しない

### 1. トピックの特定・作成 (add_topic)

transcriptを解析し、議論されたテーマを特定する。

**チェック項目:**
- 新しい議論テーマが出ていないか？
- 既存トピックの子トピックとして切り出すべきものはないか？
- トピックが広すぎて分割すべきものはないか？

**アクション:**
- 既存トピックにマッチするものがあればそれを使う
- 新しいテーマがあり、かつ記録（decision/log）が必要な場合のみ `add_topic` で作成
- 作成時は `tags` に `domain:` タグ（必須）＋内容を表すタグを付ける

### 2. アクティビティの特定・登録 (add_activity)

実装予定・作業予定をすべて登録する。ここで登録したアクティビティは、後続の資材保存（Step 3）の紐づけ先になる。

**アクティビティの判定基準:**
- 「〜を実装する」「〜を作る」「〜を追加する」
- 「〜が必要」「〜をやる」
- 決定事項から派生する実装作業

**チェック項目:**
- 記録漏れのアクティビティはないか？
- アクティビティの粒度は適切か？（1つのPRで完結する程度）
- description に十分な情報があるか？

**アクション:**
- 未登録のアクティビティがあれば `add_activity` で登録
- `tags` に `domain:` タグ（必須）＋内容を表すタグを付ける
- 関連する仕様書があれば description に参照を含める
- 明らかに完了しているアクティビティ（PRマージ済み、機能実装済みなど）があれば、ついでに `update_activity` で `completed` にする

### 3. 資材の保存 (add_material)

セッション中に生成・確認された「静的な情報」を資材として保存する。資材はセッション終了とともに揮発するため、ここで永続化する。

**materialに入れるもの（静的な情報 = 事実・成果物・スナップショット）:**
- 分析結果・調査レポート
- 設計ドラフト・仕様案
- 調査で判明した事実（仕様、挙動、制約など）
- コード（生成したコード、差分、設定ファイルの内容など）
- 比較表、一覧表

**logとの棲み分け:**
- material = それ自体は変化しない、ある時点での情報。要約せずそのまま保存する
- log = 議論の流れ・判断プロセス。material IDを参照して経緯を記録する

**アクション:**
- セッション中の生データを洗い出し、`add_material` でタグ付きで保存する（タグ必須）
- `related` で関連するアクティビティやトピックとリレーションを張る
- 保存した material ID は Step 4 の log で参照する

### 4. 決定事項・ログの記録 (add_decision / add_log)

合意が成立した内容（decision）と、議論の経緯（log）を記録する。

**決定事項の判定基準:**
- ユーザーが「OK」「それでいこう」「了解」「いいね」等で承認した
- AIの提案に対して明示的な同意があった
- 「〜で決定」「〜に決めた」等の表現がある

**ログの記録対象（議論の流れ・判断プロセス）:**
- 議論の流れ（ユーザーの発言とAIの応答の要点）
- 検討した選択肢とその比較
- 採用しなかった案とその理由
- なぜその結論に至ったかの経緯

**チェック項目:**
- 記録漏れの決定事項はないか？
- 各決定事項は適切なトピックに紐づいているか？
- decision と reason が明確に分離されているか？
- 議論の経緯がlogとして残せているか？

**アクション:**
- 未記録の決定事項があれば `add_decision` で記録
- 議論の経緯があれば `add_log` で記録
- log には Step 3 で保存した material ID を参照として含める（例: 「分析結果（material #NNN）を踏まえてA案を採用」）
- `topic_id` は関連するトピックを指定
- `tags` で内容を表すタグを付ける（省略時はtopicのタグを継承）
- **記録したdecisionの中に、今後の作業や継続議論を伴うものがあれば、対応するアクティビティを `add_activity` で作成する**（Step 2で未登録の場合）

### 5. リレーションの確認・追加 (add_relation)

記録したトピック・アクティビティ間の関連を `add_relation` で追加する。**ステップ1〜4でトピック・アクティビティの新規作成も既存エンティティへの記録もなかった場合はこのステップをスキップする。**

**判断基準:**
- 「この2つは関連している」と思ったら張る（過剰 > 不足）
- 間違ったリレーションは `remove_relation` で消せるため、迷ったら張る

**チェック項目:**
- 同じテーマを扱うtopic同士にリレーションがあるか？
- アクティビティとその背景トピックにリレーションがあるか？
- 関連するアクティビティ同士にリレーションがあるか？

**アクション:**
- `add_relation(source_type, source_id, targets=[{"type": "...", "ids": [...]}])` で追加
- 新規作成した topic / activity だけでなく、既存エンティティとの関連も確認する

### 6. 未決定の論点を記録 (add_decision)

議論が途中で終わった論点・アイデア・検討事項を記録する。**決定していなくても記録する**。

**記録すべきもの:**
- 議論に出たが結論が出ていない論点
- 「〜かも」「〜という案もある」「〜どうしようかな」
- 提案されたが承認/却下がまだのアイデア
- 次のセッションで継続すべき検討事項

**チェック項目:**
- 議論が途中で終わっている論点はないか？
- 出てきたアイデアが記録されずに消えていないか？

**アクション:**
- `add_decision` で記録（決定事項と同じテーブルで管理）
- **プレフィックスをつける**: `[議論中]` または `[未完]`
- `tags` で内容を表すタグを付ける
- フォーマット例:
  ```
  decision: "[議論中] Skillの発動条件をどう改善するか（選択肢: A案, B案, C案）"
  reason: "議論は出たが結論未定。次セッションで継続"
  ```
- 論点が大きい場合は子トピックとして切り出す

### 7. 未完の作業をアクティビティとして登録 (add_activity)

セッション中に途中で終わった話・作業がある場合、次のセッションで**このアクティビティだけ見れば文脈が取れる**レベルでアクティビティを登録する。

**対象:**
- 議論が途中で終わった話題（設計の途中、要件の詰めが途中など）
- 実装に着手したが完了していない作業
- ユーザーから依頼されたが未着手のもの

**アクティビティの書き方:**
- title: `[未完] 〜` のプレフィックスをつける
- `tags` に `domain:` タグ（必須）＋内容を表すタグを付ける
- description に**文脈を多めに**記載する。具体的には：
  - 何について話していたか
  - どこまで進んだか・何が決まったか
  - 何が残っているか・次に何をすべきか
  - 関連するトピック名・決定事項の概要

**フォーマット例:**
```
title: "[未完] sync-memoryに未完アクティビティ自動作成機能を追加"
description: "「sync-memory改善」トピックで議論中。ステップ4のdecision記録はそのまま残し、追加でadd_activityでも登録する方針で合意済み（decision A案）。残作業：SKILL.mdへのステップ追加と動作確認。"
```

**注意:**
- ステップ6の `[未完]` decision とは別に、アクティビティとしても登録する（役割が異なる：decisionは「何が未決定か」、アクティビティは「次に何をすべきか」）
- 未完の話がなければこのステップはスキップする

### 8. 棚卸し・tag notes・auto memory更新の提案

ステップ8a〜8dの結果をまとめて、**1回のAskUserQuestion**でユーザーに提案する（ステップ8e）。**8a〜8dがすべて空ならステップ8全体をサイレントスキップする。**

#### 8a. アクティビティの棚卸し

**まずスキップ判定:** ステップ0で取得したアクティビティを以下のカテゴリに照合し、**該当が1件もなければ即座にスキップする。** ユーザーにアクティビティ一覧を見せたり、「該当なし」と報告する必要はない。

**判断基準（4カテゴリ）— いずれかに該当するアクティビティがある場合のみ報告:**

（完了済みアクティビティはStep 2で処理済みのため、ここでは扱わない）

1. **重複**: 同じ目的・内容のアクティビティが複数存在する場合の片方
2. **7日以上放置**: `updated_at` から7日以上経過しているpendingアクティビティ
3. **状況変化で不要**: 前提条件が変わり、もうやる意味がないアクティビティ
4. **フェーズ移行済み**: in_progressの議論アクティビティ（intent:discuss）で、後続フェーズが既に動いているもの。以下のいずれかで検出する:
   - depends_onで下流アクティビティ（intent:design/implement）がin_progress以降
   - 同トピック内でintent:discussがin_progressかつ、intent:design/implementのアクティビティが存在する

**ユーザーの回答に応じた処理:**
- 放置アクティビティを「まだいる」と言われた場合 → `update_activity(activity_id, status="snoozed")` で寝かせる（snooze期間経過後に自動でpendingに復活する）
- 「状況変化で不要」の削除提案を却下された場合 → そのアクティビティの description 先頭に経緯を追記する（例: 「YYYY-MM-DD sync-memoryで前提変更により削除を提案したが、残すことになった。理由: ...」）

#### 8b. auto memoryへの記録提案

セッションのtranscriptを振り返り、Claudeのauto memory（MEMORY.md等）に記録すべきものがないか検討する。

**記録候補の観点:**
- 再利用可能なパターンや知見
- ハマったこと・やらかし
- ワークフローの改善点
- 重要なアーキテクチャの気づき

**記録先の判断（エージェント裁量）:**
- 短い（数行）→ MEMORY.mdに直接追記
- 長い → auto memoryの別ファイルに書き、MEMORY.mdにリンク

記録すべきものがなければ8bは空とする。

#### 8c. tag notes の更新

セッションのtranscriptを振り返り、特定のタグに紐づく教訓・運用ルールとして記録すべきものがないか検討する。
tag notes はそのタグの文脈で作業開始時にAIへ自動注入されるため、「次にこのタグで作業するAIに伝えたいこと」を書く場所である。

**対象:**
- セッション中にハマった問題で、特定のタグ文脈に共通する教訓
- 「次回このタグで作業するときに知っておくべきこと」
- 既存 notes への追記・修正が必要な情報

**auto memory との使い分け:**
- tag notes: そのタグの文脈で作業開始時に自動注入される。スコープが限定的
- auto memory (MEMORY.md): 全セッションで常に参照される。横断的な知見
- 迷ったら auto memory。tag notes は「このタグでしか使わない」と確信できるもの

**手順:**
1. セッション中に関わったタグを特定する
2. `search_tags` で対象タグを検索し、現在の notes を確認する（対象タグ名をqueryに、include_notes=Trueで呼ぶ）
3. 新規追記・修正すべき内容があればドラフトを作成する
   （既存 notes がある場合は内容を保持しつつマージ。上書き方式なので全文を書く）
4. ステップ8eでユーザーに提案し、承認後に `update_tag` で書き込む

記録すべきものがなければ8cは空とする。

#### 8d. habitsの提案

セッション中に「常に従うべきルール」が合意された場合、habitsへの登録を提案する。
habitsはcheck-in時にAIへ毎回注入されるため、tag notesよりスコープが広い。

**対象:**
- セッション中に合意した運用ルール・慣習（例: 「materialのcontentは先頭に説明を書く」）
- 繰り返し忘れがちな確認事項
- tag notesに入れるほどスコープが限定的でないもの

**tag notes との使い分け:**
- habits: 全check-inで注入。タグに依存しない横断的ルール
- tag notes: 特定タグの作業開始時に注入。スコープが限定的
- 迷ったらhabits

記録すべきものがなければ8dは空とする。

#### 8e. ユーザーに提案

8a〜8dの結果を **AskUserQuestionの別々のquestion** として提案する。各項目は独立した関心事なので、1つのquestionにまとめず分離する。

**AskUserQuestionの構成:**
- 8aが空でなければ → 棚卸し用のquestionを追加
- 8bが空でなければ → auto memory用のquestionを追加
- 8cが空でなければ → tag notes用のquestionを追加
- 8dが空でなければ → habits用のquestionを追加
- すべて空なら → ステップ8全体をスキップ

**棚卸しquestion:**
```
header: "アクティビティ棚卸し"
question: |
  **completedにするアクティビティ（確認）:**
  - id:XXX タイトル → 理由（例: PR#77でマージ済み）

  **対応が必要なアクティビティ（判断をお願い）:**
  - id:XXX タイトル → 理由（例: 7日以上放置、id:YYYと重複、前提変更で不要 など）
options: [提案通り進める, スキップ]
```

**auto memory question:**
```
header: "auto memory"
question: |
  auto memoryに記録する？
  - [記録先] 内容の概要
    例: [MEMORY.md] MCPプラグインのキャッシュは...
options:
  - label: 記録する
    markdown: |
      （記録内容のプレビューをここに入れる）
  - label: 不要
```

- 判断に迷うものは載せない（誤って消すより残すほうが安全）
- auto memoryのquestionでは `markdown` パラメータでプレビューを表示し、記録内容を事前確認できるようにする

**tag notes question:**
```
header: "tag notes"
question: |
  タグnotesを更新する？
  - [tag名] 変更内容の概要（新規追記 / 修正 / 削除）
options:
  - label: 更新する
    markdown: |
      **tag名:**
      (更新後のnotes全文)
  - label: 不要
```

- tag notes の question では `markdown` で更新後の notes 全文をプレビューする（update_tag は全文置換のため）

**habits question:**
```
header: "habits"
question: |
  habitsに登録する？（check-in時にAIへ毎回注入される）
  - {ルールの概要}
options:
  - label: 登録する
    markdown: |
      {登録するhabitsの全文}
  - label: 不要
```

**ユーザーの回答に応じて処理:**
- 棚卸し: ユーザーの指示に従って処理（completed、削除、そのまま残す等）
- auto memory: ユーザーが承認した項目をauto memoryに書き込む
- tag notes: ユーザーが承認した場合 `update_tag(tag, notes)` で書き込む
- habits: ユーザーが承認した場合 `add_habit(content)` で登録する

## 9. 完了報告

以下の形式でサマリーを報告する。件数カウントは不要。記録した内容の概要を箇条書きで列挙する。
該当がないセクションは省略する。

```
## sync-memory 完了

### トピック
- (作成したトピック名を列挙)

### アクティビティ
- (登録・完了にしたアクティビティを列挙)

### 資材
- (保存した資材のタイトルとactivity_idを列挙)

### 決定事項
- (記録した決定事項の概要を列挙)

### ログ
- (記録したログの概要を列挙)

### 棚卸し
- (処理したアクティビティを列挙)

### auto memory / tag notes
- (更新した内容の概要を列挙)

### 注意事項
- (確認が必要な事項や曖昧な点があれば記載)

### ふりかえり
まず `get_config()` を呼ぶ。`sync_disable_retrospective` が `true` の場合、このセクションをスキップする。
`roll_dice(30)` を無言で実行する（ダイスを振ることについて一切言及しない）。

- **30**: ジャックポット。`jackpot.md` を読み、再度 `roll_dice` を振ってネタを決定する。ダイスには触れず、選ばれたネタをそのまま実行する
- **1〜29**: セッションで扱った議論・作業プロセスについての所感を一言〜数行で自由に表現する（比喩、感想、なんでもOK）

ユーザーへの称賛・感謝・労いは書かない。嘘はつかない。
```

## 注意

- **未決定の論点は add_decision で記録する** - プレフィックス `[議論中]` `[未完]` をつけて区別
- 判断に迷う場合は、記録する方向で進める（後で修正可能）
- **既に記録済みのものは重複して記録しない** - ステップ0で取得した既存データと照合すること
- **現在進行中の作業がある場合は `add_activity` で登録し、`update_activity` で `in_progress` にする**
- **ステップ1〜8ではツール呼び出しの前後に判断理由・中間分析・進捗報告のテキストを出力しない。ユーザーに見せるのはステップ8dのAskUserQuestion（該当がある場合）とステップ9の完了報告のみ**
